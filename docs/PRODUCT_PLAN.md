# План продуктовых фич — family-finance-bot

> Статус: план (артефакт для согласования). Реализация — позже, поэтапно.
> Документ — источник правды по продуктовым решениям. Технические детали см. в `CLAUDE.md`.

---

## 1. Текущее состояние (MVP — готово)

Работает end-to-end и проверено на реальных чеках:

- Приём фото чека → **ОФД-first**: декод QR → запрос к ОФД (подключён **WOFD**) → LLM-парсинг эталонного текста → позиции с **GTIN+NTIN**, дата, итог, фискальный признак, подлинность. Fallback на **Gemini-Vision** (OCR), если QR нет / ОФД недоступен.
- **Классификация** по приоритету: свой каталог (GTIN/NTIN) → НК (заглушка) → ProductRule (имя) → LLM (батч, 1 запрос) → кнопки. Обучение ручных правок: по идентификатору (`product`) или по имени (`ProductRule`).
- **Уникальность** чеков (дедуп) в пределах семьи: QR-идентификаторы → фискальный номер → отпечаток.
- **Дата покупки**: из чека → спросить пользователя → дата добавления.
- **Провайдер**: Gemini (`gemini-2.5-flash` → fallback `gemini-2.5-flash-lite`), Claude опционально; умный backoff на 429.
- Команды: `/start`, `/invite`, `/report [family|week]`, `/categories`, `/learn`, `/rate`.
- Качество: 111 тестов, mypy --strict, покрытие ~86%. Ветка смержена в `main`.

## 2. Дорожная карта фич

Легенда: ⭐ приоритет · 🔜 запланировано · 💤 отложено.

### Этап 1 — Отчёты v2 + редактирование + ручное добавление ⭐🔜 (утверждается, см. §3.1, §3.3)
Перечень покупок, суммы (итог / по дням / по группам), редактирование чеков и позиций, ручное добавление покупки без чека.

### Этап 2 — Приём чеков: альбом + большой чек + PDF 🔜 (см. §3.2, §3.6)
Агрегированная обработка media-group (одна сводка), большой чек в нескольких фото, обработка **PDF-чеков**.

### Этап 3 — Управление и точность 💤
- Категории: переименование, свои категории, объединение, подкатегории.
- Каталог товаров: просмотр/правка `product`, массовое переобучение.
- Подлинность ОФД в сводке (✓), хранение БИН продавца, ссылка на копию ОФД.
- Семья/роли: статистика «кто сколько потратил», доли участников.

### Этап 4 — Расширения 💤
- **Динамика цены товара (GTIN/NTIN)** в отчётности: история цены по чекам/датам/магазинам, изменение %, алерт на подорожание (см. §3.4).
- Регулярные дайджесты (нед./мес. сводка в чат), напоминания.
- Поиск по чекам/товарам (имя, GTIN, магазин, дата, сумма).
- **Бюджеты и лимиты**: лимит на категорию/семью + уведомление.
- Мультивалютность (реальный НБК), экспорт для бухгалтерии.

### Этап 5 — Платформа: Telegram Mini App 💤 (см. §3.5)
Веб-интерфейс внутри Telegram как основной UI для отчётов, редактирования и интерактивных графиков (в т.ч. динамики цен). Привязан к этапу деплоя (нужен HTTPS-хостинг + API).

### Инфраструктура / эксплуатация 💤 (см. §4)
Деплой 24/7, платный ключ Gemini, остальные ОФД, реальные НК/НБК.

## 3. Детальные спецификации

### 3.1 Отчёты v2 + редактирование чеков (Этап 1)

**Параметры:** период + охват (я / семья).

**Периоды:** сегодня · неделя · месяц · прошлый месяц · год · произвольный диапазон (ввод двух дат).

**Режимы отчёта:**
1. **Только сумма** — итог за период (по `receipt.total_amount`).
2. **По дням** — суммы с разбивкой по датам.
3. **По группам** — суммы по категориям (по `item.total_price`).
4. **Перечень покупок** — список чеков (дата · магазин · сумма) с пагинацией; тап → карточка чека.

**Карточка чека:** магазин, дата, итог, подлинность ОФД, позиции (имя · кол-во · сумма · категория · GTIN/NTIN) + кнопки правок.

**Редактирование (покупка = чек с позициями):**
- Чек: изменить **дату**, изменить **итог**, **удалить** (с подтверждением).
- Позиция: изменить **категорию**, изменить **сумму/цену**, **удалить**.
- Правки помечаются `is_manual`; ввод чисел/дат — текстом (pending-state, как у даты); удаление — с подтверждением.

**Интерфейс:** кнопочный (inline). `/report` → Период → Режим → Охват; перечень — кнопки-чеки с пагинацией; карточка — кнопки позиций и действий.

**Изменения в коде (без миграций — поля есть):**
- repository: `sum_total`, `sum_by_day`, `sum_by_category` (есть), `list_receipts(period, page)`, `update_receipt(date/total)`, `delete_receipt`, `update_item(category/price)`, `delete_item`.
- reporter: режимы total / by_day / by_category / list.
- handlers: меню отчётов (`^rep:`), карточка чека (`^rcp:`), правки + подтверждения, pending-state.
- тесты: агрегации, пагинация, edit/delete (repo + хендлеры), роутинг.

**Под-этапы:** (1) режимы отчёта + меню периодов; (2) перечень + карточка; (3) редактирование + подтверждения.

**Вне этапа:** графики (matplotlib), экспорт CSV, разрезы по магазинам/товарам/БИН.

### 3.2 Групповая отправка чеков (альбом) + большой чек в нескольких фото

**Поведение сейчас:** альбом приходит как N сообщений (общий `media_group_id`), PTB обрабатывает последовательно → каждое фото = отдельный чек, N статусов и N сводок; на free-tier — серия вызовов; edge-case: один чек в нескольких фото → задвоение.

**A. Несколько разных чеков (альбом):** буферизовать media-group (debounce ~1–2с) → обработать пачкой → **одна** сводка «Добавлено X · дубликаты Y · ошибки Z»; последовательно или с rate-gate; учесть pending-state даты для нескольких бездатных чеков.

**B. Один большой чек в нескольких фото.**
Бот не может надёжно сам отличить «N разных чеков» от «1 чек в N фото» → **спрашиваем пользователя**.

Логика склейки зависит от наличия QR:
- **Есть QR (хотя бы на одной части):** QR ведёт в ОФД, который отдаёт **весь** чек целиком. Значит остальные фото **не нужны** — берём данные из ОФД по QR. Это основной и самый надёжный путь для фискальных чеков РК.
- **QR нет / обрезан:** отправляем **все фото одним запросом** в модель (Gemini принимает несколько изображений) с инструкцией «это один чек, объедини позиции» → один объединённый `ReceiptVisionResponse`. Дедуп — по отпечатку содержимого.

**Как пользователь это присылает (UX):**
1. **Если на чеке есть QR** — достаточно сфотографировать часть **с QR** (обычно низ чека). Бот подтянет весь чек из ОФД, дополнительные фото не требуются.
2. **Если QR нет/смазан** — отправьте все части **одним альбомом**; бот спросит: «Это разные чеки или один длинный чек?» → выберите **«Один чек»** → бот склеит позиции из всех фото.
3. Альтернатива без альбома — команда **`/longreceipt`**: бот включает режим склейки, вы шлёте части по очереди и жмёте **«Готово»**.
4. Рекомендации для съёмки: части **по порядку**, с небольшим **нахлёстом** строк, чтобы не потерять позиции на стыках.

**Реализация:** детект `media_group_id` + кнопки выбора «разные / один чек»; для «один чек» — ветка QR→ОФД либо мульти-изображение в одном LLM-запросе; команда `/longreceipt` со сбором частей и «Готово»; защита от дублей.

### 3.3 Ручное добавление покупки (без чека) — Этап 1

**Зачем:** покупки без чека (рынок, услуга, наличные) — внести вручную.

**Сценарии:**
- **Быстрый:** категория + сумма.
- **Подробный:** категория + название товара/услуги + сумма.

**Поток (`/add` или кнопка):**
1. Выбор **категории** — inline-кнопки из списка категорий.
2. (Опц.) ввод **названия** — текст; если пропустить, имя = название категории.
3. Ввод **суммы** — текст, парсинг числа.
4. (Опц.) дата — по умолчанию сегодня; меняется потом через редактирование (§3.1).

**Модель данных (без миграций):** создаётся `Receipt` без фото/QR/фискальных полей (`shop_name=null`, `photo_file_id` пустой/маркер `manual`, `fiscal_id=null`), `currency="KZT"`, `purchased_at=сегодня`, с одной `ReceiptItem` (имя, `total_price=сумма`, `quantity=1`, `category_id`, `is_manual=true`).

**Дедуп:** для ручных покупок `dedup_key=NULL` — они **не** считаются дубликатами (две одинаковые покупки в день легитимны; NULL-ключи в Postgres не конфликтуют).

**Интеграция:** ручная покупка — обычный чек из одной позиции → попадает в отчёты и редактируется наравне с распознанными (§3.1). LLM/ОФД/каталог не вызываются (категория задана пользователем); опц. можно дообучать `ProductRule`/`product` по введённому названию.

**Изменения в коде:** `handlers` — flow `/add` (ConversationHandler или pending-state: категория-кнопки → название → сумма); `repository` — переиспользуем `save_receipt_with_items`; тесты на поток и сохранение.

### 3.4 Динамика цены товара (в отчётности)

**Зачем:** видеть, как менялась цена конкретного товара со временем (инфляция, разные магазины).

**Данные (уже есть):** у каждой `ReceiptItem` — `gtin`/`ntin`/`name`, `unit_price`/`total_price`; у чека — `purchased_at`, `shop_name`. Один товар (ключ: `gtin` → `ntin` → имя) встречается в нескольких чеках → ряд точек.

**Что показываем:**
- Ряд **(дата · цена за ед. · магазин)** по выбранному товару за период/всю историю.
- **Изменение** (последняя vs первая, %), мин/макс/средняя, последняя цена.
- Опц. **алерт на подорожание** при следующей покупке («стало дороже на N%»).

**Доступ:**
- Команда `/price <запрос>` или кнопка **«История цены»** в карточке позиции (§3.1).
- Режим отчёта «По товару → динамика».

**Реализация:** repo `price_history(key, member_ids, period) -> [(date, unit_price, shop)]`; вывод — текстовый список + сводка сейчас; **интерактивный график — в Mini App (§3.5)**.

### 3.5 Telegram Mini App (будущий этап-платформа)

**Идея:** веб-приложение внутри Telegram (Web App) как основной UI для насыщенных данными сценариев, где inline-кнопки неудобны.

**Покрывает:** дашборд отчётов (периоды календарём, фильтры), **интерактивные графики** (динамика цен §3.4, расходы по категориям/дням), просмотр и **редактирование** чеков/позиций формами, перечень с пагинацией, бюджеты.

**Архитектура:** HTTP API (FastAPI рядом с ботом) поверх тех же репозиториев; авторизация через Telegram **`initData`** (валидация HMAC ключом бота → семья/участник); фронтенд (SPA) на HTTPS-хостинге; бот — точка входа (кнопка открытия Mini App) и уведомления.

**Зависимости:** этап **деплоя** (нужен сервер с HTTPS) — делать после/вместе с ним.

**Оценка удобства:** высокое для отчётов/редактирования/графиков; не нужно для простых подтверждений (их оставляем в боте). Ближняя функциональность (Этапы 1–2) реализуется в боте, чтобы не блокироваться фронтендом.

**Mini App vs полноценный web:** не «или/или». Mini App — это и есть мобильно-адаптированный web внутри Telegram; разница — во входе/дистрибуции. Делаем **один адаптивный (mobile-first) фронтенд**, публикуем **сначала как Mini App** (авторизация через `initData`, без логина), позже тот же код → полноценный web/PWA добавлением обычной авторизации (Telegram Login Widget). Второй фронтенд не нужен.

### 3.6 Обработка PDF-чеков (Этап 2)

**Зачем:** э-чеки/счета часто приходят PDF (почта, онлайн-покупки).

**Поток (тот же ОФД-first → LLM-fallback):**
1. Telegram шлёт PDF как **документ** → отдельный хендлер (`filters.Document`).
2. **Ищем QR/ссылку ОФД:** извлечь текст (`pymupdf`) → найти `oofd/consumer`-ссылку; либо отрендерить страницу → декодировать QR (zxing). Нашли → **весь чек из ОФД**.
3. **Иначе:** PDF → Gemini (нативное чтение PDF) или рендер страниц в изображения → vision. Текстовый PDF — без OCR, скан — через vision.
4. Многостраничный PDF = один чек → объединение позиций (как §3.2).

**Реализация:** хендлер документов; опц. зависимость `pymupdf` (текст/рендер/QR); дедуп — по QR/фискальному/отпечатку; тесты на текстовый и сканированный PDF.

- **Деплой 24/7:** GitHub Actions (lint→typecheck→test→build→push→deploy) + `docker-compose.prod.yml` на VPS; сейчас не настроено. Webhook вместо polling, секреты на сервере, restart-policy.
- **Платный ключ Gemini** — убрать лимит 20/мин.
- **Остальные ОФД** (Казахтелеком/oofd, Kaspi, Jusan, Transtelecom, Бухта, Вайпоинг, Кар-Тел, COMRUN, IT ANALYTICS) — через `scripts/ofd_probe.py` по образцу чека каждого оператора; добавление эндпоинта в `_ENDPOINTS`.
- **Реальный НК (`nct.py`)** — категории по GTIN/NTIN без LLM.
- **Реальный НБК** — авто-курсы валют.
- Маскирование токена в логах, метрики/алерты.

## 5. Журнал решений (согласовано с заказчиком)

- Провайдер по умолчанию — **Gemini** (`gemini-2.5-flash` + fallback `gemini-2.5-flash-lite`); Claude как опция.
- GTIN и NTIN — **отдельные поля** на позиции (требование закона).
- Каталог товаров — **глобальный**, наполняется из manual / НК / LLM; каноническое имя подменяет OCR.
- Дедуп — **в пределах семьи**.
- Источник данных по чеку — **ОФД-first**, Gemini-OCR как fallback; парсинг текста ОФД — через LLM.
- Подключение ОФД — **разведчик на Playwright (dev-only)**, рантайм только httpx.
- Отчёты: «покупка» = **чек с позициями**; редактируем **категорию, сумму/цену, дату, удаление**; интерфейс — **кнопочный**; графики/CSV/БИН — позже.
- Ручное добавление покупки (без чека): два режима — **категория + сумма** и **категория + название + сумма**; ручная покупка = чек из одной позиции, дедуп отключён (`dedup_key=NULL`).
- Большой чек в нескольких фото: бот **спрашивает** «разные/один чек»; при наличии QR — весь чек тянется из ОФД (доп. фото не нужны), иначе — склейка мульти-изображением.
- **Динамика цены товара** заложена в отчётность (§3.4); интерактивные графики — в **Telegram Mini App** (§3.5), который станет основным UI для отчётов/редактирования и привязан к этапу деплоя.
- **PDF-чеки** — поддерживаемый источник ввода (§3.6), через тот же ОФД-first → LLM-fallback.
- **Mini App vs web** — один адаптивный фронтенд: сначала Mini App (`initData`), позже тот же код → web/PWA; второй фронтенд не делаем.

## 6. Рекомендуемый порядок реализации

1. **Отчёты v2 + редактирование** (Этап 1) — утверждённый первый блок.
2. **Групповая отправка** (Этап 2).
3. **Деплой 24/7 + платный ключ** (когда нужен боевой режим).
4. **Остальные ОФД** (по мере появления образцов).
5. Управление/точность и расширения (Этапы 3–4) — по приоритету бизнеса.
